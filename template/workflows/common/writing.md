# 寫作風格 — 產出給人讀的文字時共用

← [common/README](README.md)｜[INDEX](../../INDEX.md)

產出給人讀的文字時(README、設計說明、commit 訊息、PR 描述、對外文案)動筆前先過一遍這裡。管的是**怎麼寫**,不是**做什麼**;程式碼慣例在 [conventions](conventions.md),不在這。這是**被動參考**——寫散文才取用,不是 always-on。

> 〔模板說明〕本檔是文風骨架。下面的參考作者、AI 味清單、繁簡條款都是**填好的範例**,換成你自己欣賞的文風與語言慣例。幾乎不產出散文的專案可整檔刪除(並從 common/README 移除該列)。填完後刪除所有〔模板說明〕區塊。

分兩種模式,先判斷在寫哪一種,再照那組規則走。

## 模式一:敘事／概念

**何時用**:README 開頭、設計理念、回顧、歷史脈絡、任何需要「讓人想讀下去」的段落。

**參考聲音**:
- **張大春** — 博學的機鋒,願意在一個轉折裡藏個小聰明。
- **詹宏志** — 沉思、清晰,能把技術選擇講回「人願意信任什麼」。
- **舒國治** — 留白、口語、有呼吸感,短句不堆砌。

**語氣**:有想法但不故作高深;偶爾機智;反思而不空泛;好讀。

**好的樣子**(示意):
> 每一個看似純技術的取捨,拆到最後,都是一個關於我們願意信任什麼的決定。快取要放多久、錯誤要不要重試——問的其實都是同一件事:我們有多相信下一次會跟上一次一樣。

## 模式二:技術

**何時用**:API 文件、規則說明、相容性表、測試、實作註記、教學。

**參考聲音(分情境切換)**:
- **預設:阮一峰 ＋ 林信良** — 簡潔、直接、回到本質。一句一個資訊,不修飾。先給結論,再給理由。
- **遇到容易搞混／容易踩坑處,升級成:Huli ＋ 黑暗執行緒(黑大)** — 先點破「你為什麼會混淆」,用具體例子或踩坑現場帶讀者走出來。可以幽默,可以有故事感。

**語氣**:精確、直接、實用、少裝飾。難點才多花力氣拆解,平時不囉嗦。

**好的樣子**(示意):
> 預設聲音:
> GOSUB 把下一行的位置壓入呼叫堆疊,RETURN 從那個位置繼續。堆疊是空的時候呼叫 RETURN,是錯誤。
>
> 難點切換(Huli／黑大):
> 這裡最容易搞混:你以為 `async` 會讓程式變快。它不會。它只是讓你在等的時候可以先去做別的事——變快的是「你的時間」,不是「這件事」。分清楚這兩個,後面就不會踩坑。

## 別寫成這樣(中文 AI 味清單)

> 〔模板說明〕以下針對中文;非中文專案改成你那個語言的 AI 味清單。

模型的預設壞習慣,看到就砍:

**套話開場／收尾**
- 開場:「值得一提的是」「眾所周知」「在當今……的時代」「隨著……的發展」。
- 收尾:「總而言之」「綜上所述」「總的來說」「這值得我們深思」「未來可期」。

**空洞的對仗與排比**
- 「不僅……更……」「既……又……」「與其說……不如說……」當萬用句型堆砌。
- 「它不只是 X,它是 Y」這種拿排比灌情緒的句子。

**三段式強迫症**
- 凡事「首先、其次、最後」。有邏輯順序才用,不是每段都套。

**腫脹詞彙(多半是商業黑話)**
- 賦能、打造、深耕、助力、生態、閉環、抓手、顆粒度、對齊、拉齊、心智。能用平常話就用平常話。

**過度 hedging**
- 「在一定程度上」「某種意義上」「可以說」——把話講實,不確定就說清楚不確定在哪。

**標點當戲劇效果**
- 拿冒號、破折號製造「停頓感」來裝深刻(這正是外界吐槽 LLM 的點)。

**格式雜訊**
- 每個名詞都 **粗體**、整篇都是 bullet、亂灑 emoji。強調要省著用才有力。

## 標點與節奏(中文)

- **破折號(——)少用**,別拿它當停頓的戲劇效果;多數情況一個句號或逗號就夠。
- **句長要有變化**。別每句都一樣長,長短交錯才有呼吸。
- **頓號(、)用於並列**,別用逗號代替。
- **中英文之間**統一規則(要空格就全篇空格,要不空就全不空),別混。
- 不要把英文的 em-dash 語感直接搬進中文。

## 繁簡條款

> 〔模板說明〕繁中專案才需要;簡中或其他語言專案刪掉本節。

參考的簡體作者(阮一峰等)學的是**節奏和態度,不是用字**。輸出一律繁中,別讓簡體語感滲進來。常見要轉的:

| 簡 | 繁 |
|----|----|
| 視頻 / 信息 / 網絡 / 屏幕 / 鼠標 | 影片 / 資訊 / 網路 / 螢幕 / 滑鼠 |
| 軟件 / 硬件 / 內存 / 緩存 / 字節 | 軟體 / 硬體 / 記憶體 / 快取 / 位元組 |
| 代碼 / 函數 / 對象 / 變量 | 程式碼 / 函式 / 物件 / 變數 |
| 線程 / 進程 / 默認 | 執行緒 / 行程 / 預設 |
| 用戶 / 鏈接 / 通過 | 使用者 / 連結 / 透過、藉由 |

(「數據／資料」「質量／品質」看領域,自己拿捏。)

## 文檔新鮮度

任何有意義的行為或架構變更,都要回頭檢查主要文檔集有沒有要更新。**若判斷不用改,在變更說明裡明講「文檔無需更新」**,而不是默默略過。

# DEV-GUIDE — 結構整理參考（被動 / 按需取用）

← [INDEX](INDEX.md)

這是一份**被動參考**，**不貫穿日常每個動作**——只在你要**重構 / 整理結構**（拆大檔、分類雜亂資料夾、長出新子工作流）時拿出來用。內容＝**結構整理原則**（含四級成長軌跡），docs / code / 任何資料夾通用。

> **always-on 的鐵律**（行為不變、未經確認不 push/不開新工作、各工作流具體流程在自己入口檔）常駐在 [AGENTS.md](AGENTS.md)，**不在本檔重複**——被動參考裡放 always-on 規則等於讓它失效。「要做什麼」由 [WORKFLOWS.md](WORKFLOWS.md) 派發。
> 只在**碰原始碼**時才適用的程式碼慣例與導航 index 維護鏈放在 `common/conventions.md`（由開發 flavor 包提供）。

## 結構整理原則（膨脹即拆 / 雜亂即分類）

兩個觸發、同一套方法：

- **觸發 A：單體過大** —— 某檔案、次級工作流、或原始碼大檔體量太大。**客觀門檻**（依內容性質分套訂）：
  1. **`workflows/` 流程文檔**單檔超過 **8192 bytes**（操作筆記性質，bytes 較穩定、逼它精簡）；
  2. **使用手冊 / 產品文檔**單檔超過 **300 行**（連貫敘事＋程式碼區塊多，行數較寬）；
  3. **程式碼**單檔超過 **300 行**。

  門檻是**觸發檢視的訊號**、非硬性上限：本質不可分的單體（一份完整 spec／plan、一篇連貫的調查筆記）可超標保留——其中**已完成的 spec/plan 一律移 `*/archive/` 凍結**（不留現役夾、不拆）；已凍結的 `*/archive/` 不套此門檻。
  > 〔模板說明〕門檻數字可依專案調整，但要**寫死成客觀數字**（bytes / 行數），不要「感覺太大」——客觀門檻才能讓 agent 自主觸發整理。
- **觸發 B：資料夾雜亂** —— 某資料夾底下散落各種**不同用途**的檔案，混在一起難導航。

方法（兩種觸發都照做）：

1. **按內容／職責／用途語意分**，**不是** 1/2/3/4/5 硬切等分——拆/分出來的每塊要有單一清楚的職責。
2. 放進**專屬子資料夾**（或既有領域檔）依用途歸類。
3. 子資料夾頂層留一個 **index / README** 做導航；若已有 index 則更新它。
4. 原本的頂層位置只留**精簡指標 + 指向 index 的連結**，不再塞細節。

適用場景同一套：單檔工作流膨脹到撐得起獨立資料夾、程式碼大檔超標要拆、文檔單檔過大要分檔、或一個資料夾變成什麼都丟的雜物抽屜。先分結構、補/更新 index，再讓上層只剩導航。

### 一個工作流的成長軌跡（四級）

工作流不是一開始就長成資料夾，而是**按需要逐步長出來**——每升一級都是上面「膨脹即拆」的一次套用：

1. **單檔**：一個 `.md`，**檔名即其意義**（如 `testing.md`）。把 readme（導引）、index（結構）、實際內容…**全部揉在一起**。最輕量，多數工作流停在這。
2. **資料夾＋單 README**：膨脹到需要拆 → 變資料夾，但裡面**只有一個 `README.md`**；README 仍一肩扛 readme（導引）+ guide（怎麼用）+ index（結構）全部職責（拆資料夾只是為了好擴充）。
3. **資料夾內不只 README**：README 再膨脹 → 從它**拆出獨立檔各司其職**：`INDEX.md`（頂層結構）、guide（細則，或併入本檔）、`session-log.md`（進度）、待使用者做的拆去 [WAIT_USER](WAIT_USER.md)、`gotchas.md`、內容檔…。README 退回只做「先讀我」的導引。
4. **資料夾內有子工作流**：大到內部得**再開巢狀工作流**（資料夾內若干子工作流資料夾／一個 `workflows/`），各子工作流自己又從第 1 級開始長。

往哪一級走永遠由「是否膨脹/雜亂」決定，不預先過度設計。**降級也成立**：子項消失就往回併級。
